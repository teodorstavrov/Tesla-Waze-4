## 1. СИГУРНОСТ — 8.2 / 10

### ✅ Реализирано и работи добре

**Timing-safe автентикация** (`api/_lib/admin/auth.ts`)
SHA-256 хеш на двете страни + `timingSafeEqual` — не може да се определи дължината на секрета чрез измерване на response time. Образцово изпълнение.

**Атомарен rate limiter** (`api/_lib/utils/rateLimit.ts`)
Redis `INCR` + `EXPIRE` — не съществува race window между GET и SET. Предишен наивен GET+SET позволяваше burst над лимита при едновременни заявки.
- `/api/events` POST: 5 заявки / 10 минути на IP
- `/api/ev/stations` GET: 60 заявки / минута на IP

**Защита на cron endpoint** (`api/cron/sync-stations.ts`)
Ако `CRON_SECRET` не е конфигуриран → отговор 503, не 200. Предишна версия имаше `?? ''` + `if (secret && ...)` — позволяваше unauthenticated достъп при незададена стойност.

**HSTS + CSP хедъри** (`vercel.json`)
```
Strict-Transport-Security: max-age=31536000; includeSubDomains
Content-Security-Policy: default-src 'self'; script-src 'self' 'unsafe-inline'; ...
```
Плюс `X-Frame-Options: DENY`, `X-Content-Type-Options: nosniff`, `Referrer-Policy`.

**Защита на debug endpoint** (`api/debug/ev.ts`)
Изисква `Authorization: Bearer <ADMIN_SECRET>` или `?secret=` query параметър. Неоторизирани → 401.

**HTML sanitize на event descriptions** (`api/events/index.ts:102`)
```ts
description?.replace(/<[^>]*>/g, '').trim().slice(0, 200) ?? null
```
Премахва HTML тагове, трима whitespace, ограничава до 200 символа.

**Generic error responses** (всички API handlers)
`console.error(...)` + `res.status(500).json({ error: 'Internal server error' })` — не се изтичат stack traces или вътрешни детайли до клиента.

---

### ⚠️ Оставащи рискове

**[MEDIUM] Непълна HTML sanitization** — `api/events/index.ts:102`
Regex `/<[^>]*>/g` пропуска:
- `javascript:` в href атрибути (без таг)
- HTML entities (`&lt;script&gt;`)
- Nested/malformed tags (`<scr<script>ipt>`)

Докато данните се ренедерират само в React (което ескейпва по default), рискът е нисък — но при промяна в рендеринга може да стане проблем.
**Препоръка:** Добави валидация, която разрешава само alphanumeric + пунктуация.

**[LOW] Липса на географска валидация** — `api/events/index.ts:92`
```ts
if (!isFinite(lat) || !isFinite(lng)) { ... }  // само isFinite — не проверява дали е в България
```
Може да се публикуват events на lat: 85, lng: 180. Не е сериозен риск, но замърсява базата данни.
**Препоръка:** `if (lat < 41 || lat > 44.5 || lng < 22 || lng > 28.7) → 400`

**[LOW] Липса на vote deduplication** — `api/events/confirm.ts`, `api/events/deny.ts`
Един IP може да изпрати произволен брой confirm/deny заявки на един event. Няма per-event rate limit.
**Препоръка:** Rate limit по ключ `events:vote:{eventId}:{ip}` — напр. 3 гласа / час / IP / event.

**[LOW] Debug endpoint връща station sample** — `api/debug/ev.ts:49`
Зад auth guard е, но ако `ADMIN_SECRET` е слаб, изтичат координати + source на станциите.

**[INFO] `unsafe-inline` в CSP** — `vercel.json`
Необходимо за Vite runtime injection. Не може да се премахне без nonce-based CSP, което изисква сървърно-генерирани отговори. Приемливо за текущата архитектура.

## 2. ПРОИЗВОДИТЕЛНОСТ — 8.6 / 10

### ✅ Реализирано и работи добре

**Три нива кеширане за EV станции:**
1. In-memory кеш (5 минути) — `api/_lib/cache/memory.ts`
2. Redis snapshot (седмично от cron) — `api/_lib/db/stationDb.ts`
3. Live fetch от 3 доставчика — fallback при студен Redis

**In-flight deduplication** — `api/ev/stations.ts:36`
```ts
const _inflight = new Map<string, Promise<StationsApiResponse>>()
```
N едновременни заявки към студен кеш → само 1 реален fetch; останалите await същия Promise.

**15s кеш за GET /api/events** — `api/events/index.ts:30`
100 req/min при 50 потребители → ~4 Redis reads/min (≈98% cache hit rate).

**Bbox quantization (0.5°)** — `api/_lib/utils/bbox.ts`
Леки разлики в bbox при скролване → същия кеш ключ → избягват се дублирани API заявки.

**Stale-while-revalidate на frontend** — `src/features/ev/evStore.ts`
Веднага сервира кешираните данни, refresh-ва в background. Потребителят не чака.

**localStorage fallback (24h)** за EV станции
При offline или грешка → показва данните от последния успешен fetch.

**Module-level stores (не React state)** за GPS
`gpsStore`, `followStore`, `alertEngine` — GPS tick на ~1Hz не предизвиква React re-renders.

**2s throttle на alertEngine** — `src/features/audio/alertEngine.ts:77`
```ts
if (now - _lastScanAt < SCAN_THROTTLE_MS) return  // SCAN_THROTTLE_MS = 2000
```
Proximity check при всеки GPS tick (1Hz) → само на всеки 2 секунди.

**5s reroute cooldown** — `src/features/route/routeStore.ts`
```ts
if (now - _lastRerouteAt < REROUTE_COOLDOWN_MS) return  // 5000ms
```
При GPS jitter край route deviated threshold не се тригерират верижни reroute.

**Lazy loading на modal компоненти** — `src/app/App.tsx`
`ReportModal`, `Onboarding`, `OfflineToast` → отделни chunks, не се зареждат при initial paint.

**Preconnect / dns-prefetch** — `index.html`
```html
<link rel="preconnect" href="https://router.project-osrm.org" />
<link rel="dns-prefetch" href="https://overpass-api.de" />
```
4 external доставчика → DNS resolution и TCP handshake стартират преди първата заявка.

**Service Worker с tile кеш** — `public/sw.js`
- Tile кеш: cache-first, 30-дневен TTL, max 1500 tiles
- `/api/ev/stations`: stale-while-revalidate
- При offline → 1×1 PNG fallback за tiles, `{ stations: [] }` за API

---

### ⚠️ Оставащи узки места

**[MEDIUM] Няма per-provider timeout race** — `api/ev/stations.ts:132`
`Promise.allSettled([tesla, ocm, overpass])` — ако Tesla Overpass query взема 8s, целият response чака. `fetchWithTimeout` е наличен, но при бавен provider → латентността се прехвърля директно на клиента.
**Препоръка:** Добави race с `Promise.race([fetchPromise, delay(3000)])` и serve partial results.

**[LOW] O(n²) deduplication** — `api/_lib/merge/stations.ts`
При 800 станции: ~640,000 сравнения. На сегашния обем е незабележимо (~5ms). При >5,000 станции ще стане бавно.
**Препоръка:** Grid-based spatial index при scale.

**[LOW] FIFO eviction в memory cache** — `api/_lib/cache/memory.ts:36`
```ts
const oldest = store.keys().next().value  // FIFO, не LRU
```
Горещи ключове могат да бъдат evict-нати преди студените. При MAX_ENTRIES=500 е без значение.

**[LOW] Слаба unchanged detection в cron** — `api/cron/sync-stations.ts:86`
```ts
existing[0]?.id === stations[0]?.id && existing[existing.length-1]?.id === stations[stations.length-1]?.id
```
Ако данните са същите, но пренаредени → може да се пропусне write. Практически рядко.

## 3. НАТОВАРВАНЕ ПРИ 50 ПОТРЕБИТЕЛИ

### Симулация: 50 едновременни потребители, карта на България

**GET /api/ev/stations** (~50 req/10 min при refresh):
- Warm Vercel instance: 100% cache hit → ~0.5ms response, 0 Redis reads
- Cold start: in-flight dedup → 1 fetch × 3 providers → N потребители await същия Promise
- **Резултат: ✅ Отлично**

**GET /api/events** (~100 req/min):
- 15s кеш → ≈4 Redis reads/min при 100 req/min
- Cache hit: ~0.5ms; cache miss: ~30ms Redis read
- **Резултат: ✅ Отлично**

**POST /api/events** (рядко — 1-2 пъти/час):
- Rate limit: 5 reports / 10 min / IP → защита от spam
- Race condition при едновременни writes: документирано и приемливо
- **Резултат: ✅ Добре**

**GPS rendering на frontend:**
- gpsStore + followStore: module-level, без React re-renders
- HeadingAvatar + RouteLayer: директен Leaflet API
- alertEngine throttle: scan на всеки 2s
- **Резултат: ✅ Отлично**

**Tile зареждане:**
- Service Worker кеш: cache-first
- 50 потребители × средно 20 нови tiles = 1000 tile заявки/сесия
- При загрята SW кеш: 0 мрежови заявки
- **Резултат: ✅ Отлично**

**Потенциален проблем при NAT:**
Rate limit на `/api/ev/stations` е 60 req/min/IP. Ако 50 потребители са зад NAT (напр. корпоративна мрежа с общ IP) → може да се задейства лимитът. Малко вероятно за Tesla (всяка кола има собствен IP).

### ✅ Силни страни

**Module-level stores вместо React Context**
`gpsStore`, `eventStore`, `evStore`, `routeStore`, `followStore`, `alertEngine` — обикновени JS модули с pub/sub. GPS tick от 1Hz не предизвиква каскадни re-renders.

**Imperative Leaflet pattern**
`HeadingAvatar`, `RouteLayer`, `EvMarkerLayer`, `EventMarkerLayer` — директен Leaflet API без React DOM. Критично за Tesla browser performance.

**Три-нивова кеш стратегия**
`in-memory → Redis snapshot → live fetch` — всяко ниво служи като fallback.

**TypeScript strict mode навсякъде**
`"strict": true`, `"noUnusedLocals": true`, `"noUncheckedSideEffectImports": true`.

**Service Worker offline-first**
Tile cache + stations fallback → работи при временна загуба на интернет.

**queueMicrotask за React error #310 prevention**
В `routeStore._startGpsTracking` — предотвратява nested React update по време на `gpsStore._posListeners.forEach`.

---

### ⚠️ Архитектурни слабости

**[MEDIUM] Дублирани типове frontend/backend**
`api/_lib/normalize/types.ts` и `src/features/ev/types.ts` са ръчно синхронизирани. При промяна в backend — frontend може да остане несинхронизиран.
**Препоръка:** Shared types package или code generation (tRPC / Zod inference).

**[MEDIUM] Липса на validation library**
Ad-hoc валидация (`isFinite`, `has(type)`, ръчни проверки). При разрастване — дублиращ се и лесно пропускащ се.
**Препоръка:** Zod schema на входа на всеки handler.

**[LOW] Redis без retry logic**
При transient Redis грешка → immediate failure. Без exponential backoff retry.

**[LOW] In-memory event store без persistence**
При cold start без Redis → всички events се губят. Документирано ограничение.

**[INFO] SupportModal не може да се lazy load**
`LeftControls.tsx` импортира `openSupportModal` статично → bundler не може да code-split модула.

## 9. ЗАКЛЮЧЕНИЕ

**TesRadar е добре изграден, production-ready проект с ясна архитектура и сериозно внимание към производителността на Tesla браузъра.**

След последното хардениране сигурността скочи от 5.5 на 8.2/10. Оставащите рискове са предимно низко-приоритетни edge cases (vote spoofing, географска валидация) — не критични за текущия scale.

**Главното, което липсва за пълна production зрялост:**
1. Тестове (unit + integration)
2. Error monitoring (Sentry)
3. Zod validation на API входове

Без тях промените се валидират само чрез deploy и ръчно тестване — работи за малък проект, но носи риск при бъдещи рефактори.
