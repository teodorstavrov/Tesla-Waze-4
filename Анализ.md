## СЕКЦИЯ 1 — АНАЛИЗ НА СИГУРНОСТТА

### ПОТВЪРДЕНИ ПРОБЛЕМИ

---

#### [КРИТИЧЕН] CRON_SECRET bypass при неконфигуриран env var
**Файл:** `api/cron/sync-stations.ts:37`

```ts
const secret = process.env['CRON_SECRET'] ?? ''
if (secret && headerAuth !== secret && querySecret !== secret)
```

Ако `CRON_SECRET` не е конфигуриран, `secret` е `''` (falsy), `if` short-circuits и ВСЯКА заявка без автентикация задейства пълен fetch за България от трите провайдъра (Tesla, OCM, Overpass). Изчерпва OCM API квота и натоварва Overpass.

**Поправка:**
```ts
const secret = process.env['CRON_SECRET']
if (!secret) { res.status(503).json({ error: 'CRON_SECRET not configured' }); return }
if (headerAuth !== secret && querySecret !== secret) { res.status(401).json({ error: 'Unauthorized' }); return }
```

---

#### [ВИСОК] `/api/debug/ev` е публично достъпен и разкрива Redis архитектура
**Файл:** `api/debug/ev.ts`

Endpoint-ът е маркиран `// NOT protected` и връща Redis ключови имена, структура на данните, брой станции, първото station ID и hint за cron trigger URL. Всеки може да открие вътрешната Redis схема.

**Поправка:** Добави ADMIN_SECRET проверка:
```ts
if (req.query['secret'] !== process.env['ADMIN_SECRET']) {
  res.status(401).json({ error: 'Unauthorized' }); return
}
```

---

#### [ВИСОК] `X-EV-Debug` header разкрива вътрешно състояние в production
**Файл:** `api/ev/stations.ts:79,119`

Всеки production response показва: data source (`redis`/`live`), здраве на провайдъри, вътрешно timing, Redis sync timestamps.

**Поправка:** Скрий зад `NODE_ENV !== 'production'` или премахни изцяло.

---

#### [СРЕДЕН] Липсва rate limiting на `/api/ev/stations`
**Файл:** `api/ev/stations.ts`

Events POST има rate limit (5/10мин). Stations GET — няма. Един IP може да пуска хиляди заявки в минута, форсирайки live provider fetches и изчерпвайки OCM квота.

**Поправка:** Добави `rateLimit(ip, 'stations', 60, 60)` (60 req/мин) в началото на handler-а.

---

#### [СРЕДЕН] Rate limiter използва GET+SET вместо атомарен INCR — race condition
**Файл:** `api/_lib/utils/rateLimit.ts:25-31`

```ts
const count = (await redis.get<number>(key)) ?? 0
if (count >= limit) return false
await redis.setWithExpiry(key, count + 1, windowSecs)
```

Две едновременни заявки прочитат `count=4` (limit=5), двете минават, двете записват `count=5`. Ефективният burst limit е `limit × concurrent_requests`.

**Поправка:**
```ts
const count = await redis.cmd(['INCR', key])
if (count === 1) await redis.cmd(['EXPIRE', key, windowSecs])
if (count > limit) return false
return true
```

---

#### [СРЕДЕН] Admin auth използва plain string comparison — timing oracle
**Файл:** `api/_lib/admin/auth.ts:10`

```ts
return header === `Bearer ${secret}`
```

JavaScript `===` на strings short-circuits при първия различен байт, разкривайки броя правилни prefix символи чрез response time.

**Поправка:**
```ts
import { timingSafeEqual } from 'crypto'
const a = Buffer.from(header)
const b = Buffer.from(`Bearer ${secret}`)
if (a.length !== b.length) return false
return timingSafeEqual(a, b)
```

---

#### [СРЕДЕН] Error details се показват в 500 responses
**Файл:** `api/events/index.ts:44,89`

`errorMessage(err)` връща raw `.message`, което може да включва Redis connection strings или вътрешни детайли.

**Поправка:** Логвай пълната грешка сървърно, върни generic съобщение на клиента:
```ts
console.error('[events] handler error:', err)
res.status(500).json({ error: 'Internal server error' })
```

## СЕКЦИЯ 2 — АНАЛИЗ НА ПРОИЗВОДИТЕЛНОСТТА

**Оценка: 7.5 / 10 | Tesla-browser: 8 / 10**

---

### СИЛНИ СТРАНИ

**1. Три-ниво кеширане на станции е отлично.**
In-memory (5мин) → Redis snapshot (weekly cron) → live fallback. Happy path (Redis warm) = ~10ms. In-memory hit = <1ms.

**2. GPS пътят е изцяло imperative — нула React renders на tick.**
`gpsStore` използва plain pub/sub. `HeadingAvatar` мутира DOM директно. `MapShell` вика `map.panTo()` директно. Правилно за Tesla — 1Hz GPS без React overhead.

**3. `queueMicrotask` декапьлва GPS→route store updates.**
Предотвратява nested `_emit()` calls по време на React render phase. Правилна поправка за React error #310.

**4. `preferCanvas: true` на Leaflet.**
Намалява DOM node count значително. Event markers и EV cluster circles се рендерират на `<canvas>` вместо индивидуални SVG/DOM елементи.

**5. Tile layer crossfade използва double-RAF + CSS transition.**
Без flicker при theme/satellite toggle.

**6. Route layer използва директен Leaflet API.**
`RouteLayer.tsx` никога не причинява React re-renders. Polyline segments се заменят imperatively само при промяна на маршрута.

---

### СЛАБИ СТРАНИ

**1. Няма reroute debounce.**
`routeStore.reroute()` се вика веднага при GPS отклонение >200м. На криволичещ път може да стреля 3-4 пъти за 10 секунди, всеки задействайки 300-800ms OSRM call.

**2. `alertEngine` сканира на 1Hz с пълен station scan.**
Изчислява Haversine разстояния от текуща GPS позиция до ВСЯКА станция. С 300-500 станции в паметта = ~500 изчисления/секунда — вероятно #1 main-thread hotspot при шофиране.

**3. Всичко JS се зарежда наведнъж — без code splitting.**
Admin UI, event reporting, route planning — всичко в един 80KB gzip JS bundle.

**4. `findStationsAlongRoute` се изпълнява при всяка промяна на маршрут + станции.**
Итерира всички polyline точки × всички станции. При 50km маршрут (500+ точки) и 300 станции = 150K итерации.

**5. EV marker HTML strings използват template literals.**
`makeStationIcon()` строи HTML strings per marker → `innerHTML` assignment в Leaflet. При 300+ markers е non-trivial при първо рендериране.

---

### НАЙ-СКЪПИ ПОДСИСТЕМИ (класирани)

| Ранг | Подсистема | Цена | Честота |
|------|-----------|------|---------|
| 1 | `alertEngine` station scan | ~0.5ms | Всяка 1с |
| 2 | `findStationsAlongRoute` | ~2ms | При промяна на маршрут |
| 3 | EV DivIcon HTML generation | ~5ms | При първо зареждане |
| 4 | OSRM route fetch | 300-800ms | При навигация/reroute |
| 5 | Leaflet tile fetch | Network-bound | При pan/zoom |

---

### ТОП 5 ЛЕСНИ ПОБЕДИ

1. `<link rel="preload" as="script">` в `index.html` — безплатни ~200ms
2. Премахни `X-EV-Debug` header в production — нулева цена, сигурност
3. Добави 2s debounce на `routeStore.reroute()` — 5 реда код, спира route flicker
4. Намали `alertEngine` interval от 1000ms на 2000ms — наполовина CPU
5. Добави `CRON_SECRET` empty-string guard — 3 реда, затваря потвърдена уязвимост

---

### ТОП 5 КРИТИЧНИ ПОПРАВКИ ЗА ПРОИЗВОДИТЕЛНОСТ

1. **Атомарен rate limiter INCR** — предотвратява burst abuse изчерпващ OCM квота
2. **alertEngine spatial index** — bucket станции по 0.5° grid, сканирай само nearby bucket (500→10 проверки)
3. **Reroute debounce** — елиминира route flicker на криволичещи пътища
4. **Lazy-load на route planning и event reporting** — ~30% по-малко initial parse
5. **CSP header** — намалява XSS ескалационен риск

---

### ТОП 5 TESLA-BROWSER ПОДОБРЕНИЯ

1. `<link rel="preload">` за main JS bundle в `index.html`
2. Намали `alertEngine` interval на 2000ms
3. Добави `will-change: transform` на bottom dock
4. Избягвай `backdrop-filter: blur()` на елементи с честа репaint
5. `requestIdleCallback` за non-urgent работа като `findStationsAlongRoute`

## СЕКЦИЯ 3 — СИМУЛАЦИЯ НА 50 ПОТРЕБИТЕЛИ

**Капацитетна оценка: 7 / 10**
**Вердикт: Приемливо за бета, рискове при sustained 50 concurrent**

---

### МОДЕЛ НА НАТОВАРВАНЕ

- 50 потребители, всеки с отворена карта в Tesla browser при шофиране
- Всеки поллва events на ~30с (frontend fetch interval)
- Всеки зарежда станции веднъж при стартиране, после при pan
- ~20% от потребителите задействат route calculations (OSRM)
- ~5% подават доклад за 10-минутен прозорец

---

### ОЧАКВАНА REQUEST RATE

| Endpoint | Rate на потребител | 50 потребители |
|---|---|---|
| `GET /api/events` | 2/мин | 100 req/мин |
| `GET /api/ev/stations` | 0.5/мин | 25 req/мин |
| OSRM route | 0.1/мин | 5 req/мин |
| `POST /api/events` | 0.02/мин | 1 req/мин |
| Tile fetches | 10-40/мин (Carto CDN) | 500-2000 req/мин |

---

### КАКВО ИЗДЪРЖА

**Tile fetching:** Изцяло на Carto/Esri CDNs. Нулево натоварване на backend-а. Скалира до 500+ потребители тривиално.

**Station data (warm cache):** При топъл Redis + in-memory cache, station заявките = ~2ms. 25 req/мин е пренебрежимо.

**Event reports (POST):** Rate limited до 5/10мин на IP. Спам-устойчиво.

---

### КАКВО ДЕГРАДИРА ПЪРВО

**1. `/api/events` GET — без кеширане, директен Redis read при всяка заявка.**
100 req/мин = 1.7 req/сек. Upstash Redis безплатен tier: ~10k команди/ден = 7 команди/мин средно. 100 req/мин е 14x над безплатния tier. Платен tier се справя, безплатен — изчерпва се за часове.

**2. In-memory station cache е per-Vercel-instance.**
При 50 потребители Vercel може да стартира 3-5 инстанции. Всяка с празен in-memory cache при cold start → до 5 едновременни Redis reads за едни и същи данни.

**3. Stations live fallback (Redis неконфигуриран или празен).**
При cache miss → 3 паралелни external API calls (Tesla + OCM + Overpass). При 25 req/мин с 5 Vercel инстанции → потенциално 375 external calls/мин. OCM безплатен tier: 100 req/час = 1.67 req/мин. **Ще изчерпи OCM квота под 1 минута. Най-опасен сценарий.**

**4. Публичен OSRM routing latency.**
Публичният OSRM може да отнеме 1-3с при European peak load. 5 едновременни route requests → visible "calculating..." при шофиране.

---

### ПРЕПОРЪКИ ЗА СТАБИЛНИ 50 ПОТРЕБИТЕЛИ

1. **Добави Redis event caching (30с TTL)** — 100 req/мин → ~2 Redis reads/мин
2. **Добави rate limit на `/api/ev/stations`** — предотвратява cache-miss storms
3. **Увери се, че Redis е конфигуриран преди launch** — документирай и alert при липса
4. **Self-host или платен OSRM** (Openroute Service) — елиминира public OSRM latency
5. **Vercel KV или shared cache** — споделен in-memory cache между инстанции

---

### КРИТИЧНИ ТОЧКИ

**Endpoint, най-вероятно да падне пръв:** `GET /api/ev/stations` в fallback mode → изчерпва OCM API key.

**Frontend подсистема, най-вероятно UX bottleneck:** `alertEngine` interval scan при дълги сесии на Tesla browser.

### ФИНАЛЕН ВЕРДИКТ

**Готов за ограничена бета. Нуждае се от hardening преди 50-потребителска Tesla употреба.**

Приложението има правилна архитектура за Tesla browser (imperative GPS/map updates, canvas rendering, pub/sub без React re-renders). Производителността е добра при единичен потребител. При 50 потребители главният риск е OCM API quota exhaustion при Redis cache miss storm — сценарий, напълно предотвратим с конфигурация на Redis и rate limiting. Сигурността е структурно добра, но има 2 потвърдени high/critical проблема (CRON bypass + debug endpoint) изискващи поправка преди публично излагане.
